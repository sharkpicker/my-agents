# 工作流优化设计文档

> 日期: 2026-06-29
> 状态: 设计中
> 目标: 在保证分析精度不变的前提下，通过公共模块抽离、流程精简、Prompt优化，大幅降低 Token 消耗和运行时间

---

## 1. 背景与目标

### 1.1 当前问题

| 问题 | 表现 | 影响 |
|------|------|------|
| 数据重复拉取 | 多个分析师各自拉取相同数据（如 global-news、fund info） | Token浪费、速度慢 |
| 单标的流程冗余 | 组合场景每只标的独立跑完整研判流程（辩论/研究/交易/风控/组合经理） | 调度次数翻倍，约50%冗余 |
| Prompt冗长 | 每次Task调用重复传角色定义、数据命令、输出格式 | Token消耗大 |
| 编号混乱 | Step 0.5 / 1.1 / 2.6 / 5.5 等编号不直观 | 维护成本高 |

### 1.2 优化目标

| 指标 | 目标 |
|------|------|
| Token消耗（组合场景） | 减少 60-70% |
| 调度次数（组合场景） | 减少 50% |
| 分析精度 | 保持不变 |
| 代码可维护性 | 显著提升 |

## 2. 优化一：数据预拉取模块

### 2.1 核心思路

在分析师并行之前，由主对话一次性拉取所有公共数据，存入共享目录。
分析师直接读取已保存的数据，不再各自拉取。

### 2.4 智能增量拉取（替代预拉取清单）

**不新增任何元数据文件**，直接查本地文件判断是否需要拉取。

#### 判断逻辑

```
主对话：需要什么数据？
  - 近2年K线数据
  - 近3个月新闻
  - 今天的市场情绪（global_news）

    ▼

检查本地文件
  - kline_2024-06-29_2026-06-27.csv
    → 已有，但只到昨天，需增量1天
  - news_2026-03-29_2026-06-27.md
    → 日期范围够用，直接用
  - global_news.md
    → 文件更新时间 2026-06-27 14:00
    → 距今超过24小时，需刷新

    ▼

生成增量拉取命令清单
  - kline 000001 --start 2026-06-28
  - global-news
```

#### 数据时效性规则

| 数据类型 | 有效期限 | 判断依据 |
|---------|---------|---------|
| K线/净值 | 近2年/1年 | 文件内日期范围 |
| 新闻 | 近3个月 | 文件内日期范围 |
| global_news | 24小时 | 文件修改时间 |
| hot_stocks | 24小时 | 文件修改时间 |
| northbound | 24小时 | 文件修改时间 |
| 基本面/基金信息 | 7天 | 文件修改时间 |
| 季报/年报数据 | 无过期 | 文件内日期范围 |

#### 存储结构（不变）

```
data/
  stocks/
    <code>/
      kline_*.csv              # 复用现有路径
      fundamentals.txt
      news_*.md
      global_news.md
      # ... 其他现有文件
  funds/
    <code>/
      nav_*.csv
      fund_info.txt
      # ... 其他现有文件
```

**无新增任何文件**，完全复用现有存储结构。

### 2.5 主对话增量拉取流程

```python
def ensure_data_ready(code, data_type, requirements):
    """确保本地有满足需求的数据，不满足则增量拉取"""
    local_files = list_files(f"data/stocks/{code}/")
    
    for req in requirements:
        if req.type == "kline":
            # 检查日期范围是否满足
            existing = find_file(local_files, f"kline_*.csv")
            if existing and covers_date_range(existing, req.start, req.end):
                continue  # 已有足够数据
            else:
                # 增量拉取
                run_command(f"kline {code} --start {req.start} --end {req.end}")
        
        elif req.type == "global_news":
            # 检查修改时间
            existing = find_file(local_files, "global_news.md")
            if existing and file_age_hours(existing) < 24:
                continue  # 24小时内有效
            else:
                run_command("global-news --limit 30")
```

### 2.6 分析师视角（不变）

分析师不需要知道"预拉取"的概念：
- 直接读取本地现有文件
- 如果发现数据缺失或不满足需求，自行补充拉取（回退机制）

## 3. 优化二：组合工作流单标的流程精简

### 3.1 核心思路

组合场景下，每只标的只跑 7 分析师（做研究），研判流程（辩论/研究/交易/风控/组合经理）全部上移到组合层面统一执行。

### 3.2 优化前后对比

**优化前（5只基金组合）**：

```
Step 1: 5只 × 7分析师 = 35次
Step 1.3: 5只 × 8个研判subagent = 40次
Step 2-7: 组合层面 = 8次
总计: 83次
```

**优化后（5只基金组合）**：

```
Step 4: 5只 × 7分析师 = 35次
Step 7-12: 组合层面 = 6次
总计: 41次
```

**减少 50.6% 的调度次数**。

### 3.3 组合层面研判流程

组合层面的subagent读取所有标的的分析师报告，做整体研判：

| 步骤 | 角色 | 输入 | 输出 |
|------|------|------|------|
| Step 7 | 多空辩论 | 所有标的的7分析师报告 + 组合分析报告 | 多空论点 |
| Step 8 | 研究经理 | 同上 + 辩论记录 | 整体投资计划 |
| Step 9 | 交易员 | 同上 + 投资计划 | 整体交易方案 |
| Step 10 | 候选基金推荐 | 同上 + gap分析 | 推荐候选基金 |
| Step 11 | 风控审查 | 同上 + 推荐结果 | 风控意见 |
| Step 12 | 组合经理 | 全部材料 | 最终报告 |

### 3.4 单标的场景保持不变

A单股票 / B单基金：只有1只标的，没有"组合层面"，完整流程保留不变。

## 4. 优化三：Subagent Prompt 精简

### 4.1 核心思路

角色定义完全收归 `agents/<name>.agent.md`，调用prompt只传必要参数（读什么文件、用什么数据、输出到哪里）。

### 4.2 优化前后对比

**优化前**（约300-500字）：

```
你是一位基金市场分析师(fund-market-analyst)。
你的核心任务是通过分析基金历史净值走势、各阶段收益率、同类排名变化，评估基金的相对强弱与趋势健康度。

请：
1. 读取 agents/fund-market-analyst.agent.md，严格按照输出格式完成报告
2. 对基金 001717 拉取数据：
   - python -m data_tools.cli fund performance 001717
   - python -m data_tools.cli fund nav 001717 --start 2025-06-29 --end 2026-06-29
3. 数据保存到 data/funds/001717/
4. 报告保存到 reports/2026-06-29/fund/001717_market.md
5. 返回核心要点摘要
```

**优化后**（约100-150字）：

```
角色: fund-market-analyst
标的: 001717（工银前沿医疗股票A）
数据目录: data/funds/001717/shared/
输出路径: reports/2026-06-29/fund/001717_market.md

请:
1. 读取 agents/fund-market-analyst.agent.md 获取角色定义和输出格式
2. 读取数据目录下的所有数据文件
3. 完成分析报告并写入输出路径
4. 返回契约格式(summary/detail_path/evidence)
```

### 4.3 通用调用模板

所有subagent统一使用以下模板：

```
角色: <agent-name>
标的: <code>（<name>）  # 组合场景可传标的列表
数据目录: <data_dir>
输出路径: <output_path>
上下文: <额外说明，如组合类型、风险等级等>

请:
1. 读取 agents/<agent-name>.agent.md 获取角色定义和输出格式
2. 读取数据目录下的相关数据
3. 完成分析/研判并写入输出路径
4. 返回契约格式(summary/detail_path/evidence)
```

### 4.4 适用范围

**所有通过Task工具调用的subagent**：
- 7大股票分析师
- 7大基金分析师
- 多头/空头研究员
- 研究经理
- 交易员
- 激进/保守/中立风控
- 组合经理
- 组合分析师
- 数据质量审计员
- 候选基金推荐员

## 8. 风险与回退

### 8.1 质量风险
- 风险：组合层面研判可能不如单标的细致
- 应对：先在测试环境对比优化前后的报告质量，确认无显著下降再上线
- 回退：保留旧工作流文档，必要时可快速回退

### 8.2 数据风险
- 风险：预拉取数据可能不完整，导致分析师缺数据
- 应对：分析师subagent仍可自行补充拉取缺失数据
- 回退：数据预拉取失败时，分析师自动回退到自行拉取模式

### 8.3 兼容性风险
- 风险：已有测试/脚本依赖旧编号
- 应对：逐步更新，保留旧编号别名（可选）
- 回退：编号重命名为纯文档变更，不影响功能
